# 2026-03-31 - PM Agent Memory

## Harness Engineering：2026 年 AI 工程新范式

### 核心概念
- **定义**：Harness 是 AI 的"操作系统"，是一套围绕 AI Agent 构建的约束、反馈与控制系统
- **公式**：Agent = Model + Harness
- **核心理念**：Humans steer, Agents execute（人类掌舵，智能体执行）

### 架构类比
```
模型（Model）= CPU（提供算力）
上下文（Context）= RAM（临时记忆）
Harness = 操作系统（约束、控制、反馈）
```

### 三次 AI 工程进化
| 年份 | 范式 | 解决什么问题 |
|------|------|-------------|
| 2022-2024 | Prompt Engineering | 对 AI **说什么** |
| 2025 | Context Engineering | 给 AI **喂什么** |
| 2026 | Harness Engineering | 让 AI 在**什么环境**里做事 |

### 关键区别
- **Context** 给 AI "知识"
- **Harness** 给 AI "**约束**"

---

## Harness 的 5 大核心能力

### 1. 精准信息装配
- 不是喂"百科全书"，而是给"地图"
- 上下文工程 + RAG 智能调度
- 分层知识架构 + 渐进式注入

### 2. 工具接口 + 沙盒化
- 统一工具调用格式
- 最小权限 + 隔离环境
- 杜绝删库等灾难性操作

### 3. 刚性流程约束
- 架构规范 → 可执行的硬规则
- CI/CD 门禁自动拦截违规代码
- "规则死，执行活"

### 4. 外部化记忆 + 状态追踪
- 持久化长期记忆（向量库/数据库）
- 断点续跑 + 目标纠偏
- 实时任务状态机

### 5. 可观测性 + 闭环反馈
- 全链路监控决策过程
- Agent 自检 + 多 Agent 评审
- 执行-校验-纠错-优化闭环

### 核心区别：AI 原生 vs AI 化尝试

#### 传统游戏公司的"AI 化尝试"
- **本质**：在现有游戏框架中嵌入 AI 能力
- **路径**：游戏设计 → AI 增强（降本增效）
- **示例**：
  - 《和平精英》接入 DeepSeek AI 队友
  - 《逆水寒》智能 NPC 系统
  - 《王者荣耀世界》AI 动态剧情生成
  - 绿洲启元 AI 编辑器
- **特点**：游戏本质不变，AI 是工具/增强功能

#### Loopit 的"AI 原生"
- **本质**：以 AI 为核心原点重新构想内容形态
- **路径**：AI 能力 → 新的内容交互范式
- **产品哲学**：
  - 不是"用 AI 做游戏"
  - 而是"AI 重新定义内容本身"
  - 用户输入一句话，AI 生成可互动的 H5 内容
- **特点**：AI 不是工具，而是内容的创作者、运行时、引擎

### 技术架构创新

#### Coding + 多模态融合
- **核心**：AI Coding 驱动多模态生成构成互动引擎
- **互动 Runtime**：为 AI 重新设计的运行时环境
  - 统一接管用户交互（点击、拖动、传感器）
  - 持续维护内容状态
  - 约束多模态生成在代码边界内
- **开发周期**：7 个月，每隔几周推翻 demo 持续重构

#### 用户交互方式
- 点击、划动、戳屏
- 摇一摇（陀螺仪）
- 吹气（麦克风）
- 摄像头识别
- 表情识别

#### 创作门槛
- **零门槛**：输入一句话，AI 自动生成点子 + 全模态互动内容
- **支持**：图像、语音、视频、3D、物理引擎、音效
- **示例**："Y2K 风格的答案之书"仅需两轮对话完成

### 市场表现
- **融资**：过去 30 天内完成两轮融资，最新估值较 1 个月前提升 6 倍
- **高鹄资本**：担任独家财务顾问
- **马斯克转发**："躲避 Grok 按钮"互动内容获马斯克点赞转发

### 产品定位洞察
- **不是**："AI 游戏平台"
- **而是**："AI 内容社区"
- **关键差异**：
  - 创作工具无法成为平台（如剪映无法成为抖音）
  - Loopit 产生网络效应
  - 通用产品，而非垂直产品
  - 用户自己探索边界

- **定位**：标准化的 brainstorm → plan → execute → review 流程
- **核心数据**：120k+ stars，MIT 协议，几乎 Claude Code 的标配
- **核心工作流**：brainstorming → writing plans → executing plans → code review
- **核心技能**：
  - Brainstorming（需求分析）
  - Git Worktrees（分支管理）
  - Writing Plans（计划制定）
  - Test-Driven Development（测试驱动开发）
  - Subagent-Driven Development（子代理驱动开发）
  - Systematic Debugging（系统化调试）
- **风格**：工程方法论，严格流程约束

### 对比总结
| 维度 | gstack | Superpowers |

| **定位** | 角色扮演系统 | 工程方法论 |

| **适用场景** | 创业公司全栈 | 专业开发团队 |
| **学习曲线** | 中等 | 中等 |
| **复杂度** | 高 | 中 |
| **实战价值** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

**推荐**：
- **选择 gstack**：如果你是创业者，需要一个人扮演整个团队
- **选择 Superpowers**：如果你是工程师，想要更可靠的代码质量

- 融到巨额资本 → 训练下一代模型 → 模型能力碾压

**护城河**：
- 算力垄断
- 技术降维打击
- 模型能力碾压

**风险**：
- 无收入支撑的 R&D

## "资本飞轮"：算力垄断的暴力美学

### 核心逻辑
> "这可能是历史上第一次，你可以将投资的美元直接转化为产出结果。"

**Martin Casado（a16z 合伙人）**：
- 只要"缩放定律（Scaling laws）"还在起作用
- 模型公司融到的每一分钱，买来的每一张 GPU
- 都能直接转化为模型能力的线性提升
- 而能力的提升，会立刻带来海量的市场需求

### 现实数据
- **OpenAI**：3 人 + 5 个月 + 0 行代码 = 100 万行代码
- **LangChain**：不改模型，仅优化 Harness → 准确率 52.8% → 66.5%（+13.7%）

### "无聊企业级软件"的护城河

**中国案例**：
- **DeepSeek**：中国版 OpenAI，降本增效
- **月之暗面、MiniMax**：积累私有数据 + 市场契合
- **腾讯、字节**：AI + 现有业务结合

**毒舌点评**：
> "无聊企业级软件"的护城河不是 AI 技术，是**客户资源 + 业务垄断**。腾讯、字节这种"无聊"公司，拥有数亿用户和深厚业务，恰恰是他们能利用 AI 增强"护城河"的原因。所谓的技术创新，只是给护城河穿了一件外衣。

---

## 终局推演：3 年内的两条路径

### 路径 A：寡头垄断（a16z 观点）

**现状**：
- **底层模型公司**（OpenAI、Anthropic）：融资最多、算力最强、模型最先进
- **应用层公司**（Character.ai、Perplexity）：找 PMF、市场验证阶段

**终局推演**：
- 如果底层模型公司能无限烧钱（OpenAI 可以）
- 他们终将通过"降维打击"所有应用层公司
- 所有应用层公司要么被收购，要么被降维打击到"AI 增强"

**中国玩家**：
- **深入**：只能跟进开源模型
- **出局**：应用层创业困难
- **机会**：提供本地化数据、私有化场景、深耕传统行业

---

### 路径 B：产品市场契合（中国创业公司的优势）

**现状**：
- **"无聊企业级软件"**：腾讯、字节、网易、恺音
- **护城河**：深厚客户资源 + 庞大 DAU + 业务垄断

**终局推演**：
- 中国应用层公司 **不依赖底层模型公司的"施舍"**
- **自主可控**：数据、用户、流程都在自己手里
- **深度集成**：AI 是工具，不是"主心"

**海外玩家**：
- **出局**：无法参与 AI 游戏、AI 原生
- **机会**：聚焦本地化场景、传统行业
